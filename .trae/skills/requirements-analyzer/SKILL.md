# 需求分析与明确化

## 功能概述

本SKILL帮助用户分析和明确化原始需求，识别需求中的问题并生成明确的需求列表。

## 使用场景

- 用户提供了原始需求文档，需要分析和明确化
- 需求中存在模糊表述、矛盾或遗漏
- 需要将需求转换为标准的Gherkin格式
- 需要生成问题清单与用户确认

## 工作流程

### Step 1: 收集原始需求

1. 如果用户提供了需求文档路径，读取该文档
2. 如果用户直接输入需求内容，记录该内容
3. 如果用户描述了需求，让用户补充文档或详细描述
4. 确保收集到完整的需求信息

### Step 2: 需求解析

解析原始需求，提取以下信息：

**识别模糊表述**：
- "可能"、"也许"、"大概"
- "支持某些功能"、"若干功能"
- "性能要够用"、"响应要快"
- "需要考虑"、"可以支持"

**识别矛盾需求**：
- 相同功能的不同描述
- 互相排斥的条件
- 不一致的参数要求

**识别不完整信息**：
- 缺少边界条件
- 缺少触发条件
- 缺少预期结果
- 缺少优先级

**识别错误**：
- 语法错误
- 逻辑错误
- 遗漏必要信息

### Step 3: 生成问题清单

为每个识别出的问题生成确认问题：

```
### 需求确认

#### 问题 1: [问题描述]
- **原始内容**: [需求原文]
- **问题类型**: [模糊/矛盾/遗漏/错误]
- **建议**: [修改建议]
- **确认**: 请确认此需求的正确描述是什么？
```

### Step 4: 输出明确需求

使用Gherkin格式输出明确的需求：

```gherkin
## Requirement: [需求名称]

### Scenario: [场景描述]
- **WHEN** [触发条件]
- **THEN** [预期结果]
- **AND** [附加条件]
```

**Gherkin格式说明**：

| 关键词 | 用途 |
|--------|------|
| Feature | 功能特性 |
| Requirement | 需求项 |
| Scenario | 场景描述 |
| WHEN | 触发条件 |
| THEN | 预期结果 |
| AND | 附加条件 |
| Given | 初始条件 |

### Step 5: 输出需求表格

生成需求跟踪表格：

```markdown
| ID | 需求名称 | 优先级 | 状态 | 备注 |
|----|----------|--------|------|------|
| REQ-001 | 需求描述 | 高 | 明确 | |
```

## 需求质量检查清单

### 完整性检查
- [ ] 每个需求都有明确的触发条件
- [ ] 每个需求都有明确的预期结果
- [ ] 需求覆盖了正常和异常场景
- [ ] 需求覆盖了边界条件

### 一致性检查
- [ ] 需求之间没有矛盾
- [ ] 需求与系统约束一致
- [ ] 需求与接口规范一致

### 可验证性检查
- [ ] 需求可以被测试验证
- [ ] 需求描述清晰无歧义
- [ ] 需求条件可判定

### 可追溯性检查
- [ ] 每个需求有唯一ID
- [ ] 需求与上层特性关联
- [ ] 需求与设计实现关联

## 输出格式

### 1. 问题清单

```markdown
## 需求分析报告

### 发现的问题

#### 问题列表
| 序号 | 类型 | 原始内容 | 问题描述 | 建议修改 |
|------|------|----------|----------|----------|
| 1 | 模糊 | "性能要够用" | 未明确性能指标 | 明确具体性能要求 |
```

### 2. 明确需求列表

```markdown
## 明确的需求

### Feature: [功能特性]

#### Requirement: [需求名称]
- **ID**: REQ-XXX
- **Priority**: [高/中/低]
- **Scenario**: [场景名]
  - **WHEN**: [触发条件]
  - **THEN**: [预期结果]
```

### 3. 需求跟踪表

```markdown
## 需求跟踪表

| ID | Feature | Requirement | Priority | Status |
|----|---------|-------------|----------|--------|
| REQ-001 | 功能A | 需求描述 | 高 | 明确 |
```

## 示例

### 输入（原始需求）

```
需要实现一个处理器核，支持RISC-V指令集
性能要尽量优化
支持中断和异常处理
功耗要低
```

### 输出（明确需求）

```gherkin
## Requirement: RISC-V指令集支持
系统应支持RISC-V RV64GC指令集。

### Scenario: 基本指令执行
- **WHEN** 处理器接收到有效指令
- **THEN** 正确解码并执行指令
- **AND** 更新相应状态寄存器

### Scenario: 指令异常
- **WHEN** 指令执行发生异常
- **THEN** 跳转到异常处理向量
- **AND** 记录异常原因到mcause寄存器

---

## Requirement: 中断处理
系统应支持中断响应和处理。

### Scenario: 外部中断响应
- **WHEN** 外部中断信号有效且中断使能打开
- **THEN** 响应中断并跳转到中断处理程序
- **AND** 保存当前PC到mepc寄存器
```

## 注意事项

1. **问题识别要全面**：不要遗漏任何模糊或矛盾点
2. **问题描述要准确**：准确描述问题的具体位置和性质
3. **建议要合理**：提供合理的修改建议
4. **保持用户参与**：让用户确认关键问题
5. **输出格式要规范**：使用标准Gherkin格式
