# OpenC910 RTU (Retire Unit) 详细设计说明书

## 1. 引言与概述 (Overview)
### 1.1 模块功能定位
RTU (Retire Unit，退休单元) 实际上也就是业界所称的 Reorder Buffer (ROB) 等管理逻辑的核心。
在以 "乱序执行，按序提交 (Out-of-Order Execution, In-Order Commit)" 为基础的超标量处理器中，RTU 是维持精确异常管理和机器架构状态强一致性的绝对守护者。
它负责收容从 IDU 分发出来的有序指令，等待各种前端和后端流水线的乱序执行结果，然后依靠一个环形序列严格要求它们按照最初拿到的顺序“退休”（将结果真切地写进体系结构）。

---

## 2. 核心二级子模块及数据流向 (Top-Level Architecture)

根据 OpenC910 `gen_rtl/rtu/rtl` 下的模型结构，其由多种状态维护队列和异常决策逻辑组成。

### 2.1 重排序缓冲主体 (ROB)
- **功能**：由多项 Entry 组成的大型环形 FIFO。当 IDU 解码指令时即在 ROB 尾部分配表项（Allocate），记录下指令类型、目的逻辑寄存器。
- **执行追踪**：各执行单元（IU, LSU, VFPU）完成时，会拿着对应的指针（如 `iid`）向 ROB 更新状态为 "完成" (Complete)。
- **按序退休 (Commit/Retire)**：RTU 的头部指针检查。只有当最老的一条指令或一组（超标量架构下每周期能并排提交多条）被标记为完成后，才认定退休，使寄存器重命名生效。

### 2.2 寄存器回收体系 (Free List Management & Retire)
- **映射恢复**：正常退休时，一条指令覆盖了特定的逻辑寄存器，RTU 通知 IDU 中 Rename 层释放该逻辑寄存器上一次挂载的陈旧物理寄存器，将其交还给空闲列表 (Free List) 以备循环分配。
- **异常快照恢复**：若 ROB 头部的指令发生异常/或存在先前的错误跳转分支，RTU 下令放弃 ROB 内所有后续年轻指令的暂存结果，将 Rename 的 Map Table 弹回（Restore）并强制指令回滚。

### 2.3 异常拦截与控制流管理 (Exception Handling)
- **陷阱汇总**：无论是 IFU 取指的 Bus Error、IDU 的 Illegal Instruction，还是 LSU 对齐失效、IU 的溢出，或者 CP0 侧传来的软外部中断，所有陷阱全部压入对应的 ROB 表项。
- **拦截确认**：必须等附带异常的指令按序走到 ROB 头部时，才能确凿地扔出精确异常 (Precise Exception)，以防它是被错误推测执行的代码。

## 3. 对外接口规格 (Module Interfaces & IO)

### 3.1 RTU 与 IDU (入口与回收端)
- **Alloc (入队)**：`idu_rtu_pst_dis_inst0_preg` 等表明 IDU 请求开辟对应的 ROB 站位及关联。
- **Release (离队)**：反馈给 IDU，告知某物理寄存器可以放回空闲池等待再分配。

### 3.2 RTU 与 所有执行单元 (IU, LSU, VFPU)
- **完成汇报**：各个运算器结束自己的运算后，提交通知 (`iu_rtu_ex2_pipe*_wb_preg_vld`)。
- **冲刷命令**：遇到错误后发送雷霆广播指令（例如 `rtu_idu_flush_fe`, 冲刷整个流水线数据）。

### 3.3 RTU 与 CP0 (提交至系统级)
- 传递真正按序发生的体系结构层级更改，如中断激活、模式换取等向特权指令单元通信。

---

## 4. 微架构亮点总结
- **兼具深度与宽度的乱序提交收口**：支持高度并发的大位宽提取。它隐藏了执行端的混沌无序，将多核或超标量系统最为艰难的一面（回退、清理、状态维护）做出了稳如磐石的封口闭环。
- **精准分支对峙**：如果判定为错误分支预测，即便在满配执行时，也能极低代价通过回溯重置快照矩阵，而非繁重地一阶阶倒播数据。
